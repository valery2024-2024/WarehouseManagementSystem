# Iteration 2 — Бізнес-логіка, Persistence та Analytics

## Реалізовані Use Cases

### 1. Управління товарами

Система дозволяє:

- додавати товари;
- переглядати всі товари;
- шукати товари;
- сортувати товари за ціною;
- фільтрувати товари з малою кількістю.

### 2. Обробка замовлень

Система підтримує:

- створення замовлень;
- автоматичне списання товару;
- обчислення загальної вартості;
- застосування знижок через Strategy Pattern.

### 3. Аналітика складу

Система надає:

- загальну вартість складу;
- аналітику замовлень;
- статистику продажів товарів;
- LINQ-фільтрацію та сортування.

---

### Persistence Layer

Реалізовано JSON persistence з використанням асинхронних файлових операцій.

Можливості:

- збереження товарів у JSON;
- завантаження товарів із JSON;
- відновлення стану застосунку при запуску;
- обробка ситуацій, коли файл відсутній.

Реалізовані класи:

- JsonProductDataStore
- IDataStore<T>.

### Бізнес-правила

Реалізовані бізнес-правила:

1. Ціна товару не може бути від’ємною.
2. Кількість товару не може бути від’ємною.
3. Кількість у замовленні повинна бути більшою за нуль.
4. Загальна сума замовлення повинна бути більшою за нуль.
5. Кількість товару на складі не може бути меншою за нуль.
6. Знижки застосовуються залежно від типу клієнта.

Бізнес-правила перевіряються:

- у domain entities;
- у service layer;
- в automated tests.

### Патерни проєктування

#### Strategy Pattern

Використовується для розрахунку знижок.

Реалізовані стратегії:

- RegularDiscountStrategy
- SilverDiscountStrategy
- GoldDiscountStrategy

Фабрика:

- DiscountStrategyFactory

Переваги:

- можна легко додавати нові типи знижок;
- бізнес-логіка залишається розширюваною;
- дотримується принцип Open/Closed Principle.

### LINQ-запити

Реалізовані LINQ-операції:

- сортування товарів за ціною;
- фільтрація товарів з малою кількістю;
- обчислення загальної вартості складу;
- генерація статистики та аналітики.

### DTO Layer

Реалізовані DTO-класи:

- CreateOrderRequest
- OrderSummaryDto
- OrderAnalyticsDto
- ProductSalesDto

DTO використовуються для відокремлення domain entities від відповідей application layer.

### Repository Refactor

Repository layer було покращено через:

- GetById(Guid id)
- Update(Product product)

Repository interface:

- IProductRepository

### Automated Testing

Реалізовані xUnit тести для:

- перевірки Product entity;
- перевірки Order entity;
- перевірки discount strategies;
- перевірки складських операцій.

Проєкт містить:

- unit tests;
- CI workflow;
- автоматичну перевірку збірки.

### Ризики та майбутні покращення

Можливі майбутні покращення:

- підтримка XML persistence;
- інтеграція з базою даних;
- розширена аналітика;
- integration tests;
- система повідомлень;
- підтримка Observer Pattern.

Можливі ризики:

- проблеми цілісності persistence;
- одночасна зміна залишків товарів;
- відновлення після пошкодженого JSON.
